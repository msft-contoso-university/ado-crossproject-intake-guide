Cross-Project Work Intake in Azure DevOps — A Workflow for the P Project

A practical, enterprise-grade blueprint for letting IT teams and business teams in other Azure DevOps Projects submit User Stories into your P team Areas so you can triage, allocate, and deliver work — while preserving 7Pace time tracking, audit compliance, and full traceability. Recommendations are grounded in Microsoft Learn documentation and curated research via the MicrosoftDocs Azure DevOps agent skill.

Key Microsoft guidance that shaped this design: Microsoft explicitly recommends using a single project to support your organization or enterprise and notes that moving work items between projects “results in the loss of associated history.” That is why our primary recommendation keeps the work inside the P project and uses Area Paths + automation to bring external teams in — rather than federating across many projects. The Problem We’re Solving

the P Project already has a healthy model for allocating work inside the P project: each IT team owns an Area Path, uses a team backlog, and cross-team work is handled by setting the User Story’s Area Path to the receiving team. The friction appears at the project boundary.

✅ What works today

  • All P IT teams share one Project — clean backlogs via Area Path
  • Stories roll up to Features and Epics across teams
  • 7Pace tracks time on any work item in the P project
  • Cross-team allocation = just switch the Area Path

❌ Where it breaks

  • Other IT teams and business teams live in different ADO Projects
  • No way for them to pick an Area Path that lives in P
  • Work requests arrive via email, Teams, or tickets — not trackable
  • P loses visibility into upstream demand and can’t roll it up to capacity plans

Goal: Give any team in the Azure DevOps organization — regardless of which Project they live in — a simple, governed path to submit a User Story that ends up in the correct P team Area, with the originating team retaining visibility and 7Pace time tracking intact.

Azure DevOps Constraints You Need to Know

Before we pick a pattern, it helps to be explicit about what Azure DevOps does and does not allow across Project boundaries. This is the reason the answer isn’t obvious — the platform enforces a hard line at the Project.

CapabilityWithin a ProjectAcross Projects (same Org)Notes
Assign a work item to an Area Path Yes No Area Paths are scoped to their owning Project. A User Story cannot point at an Area Path in another Project.
Link work items (Related / Parent / Child / Predecessor) Yes Yes Cross-project links are fully supported inside the same organization.
Cross-project queries Yes Yes Flat and directed-link queries can span multiple projects in one org.
Move a work item to another Project n/a Yes Native feature. Preserves ID history, comments, attachments. Requires the source and target projects to use compatible processes and the user to have Move work items out of this project + Edit work items in this node on the target.
Add custom work item types / states Yes Per process Only Inherited Processes in Azure DevOps Services. A process is shared by any project that uses it.
Service Hooks / Webhooks Yes Yes Per-project subscriptions; can fire on cross-project events.
Power Automate — Azure DevOps connector Yes Yes A single flow can read from Project A and create/update work items in Project B, setting Area Path at creation time.
7Pace Timetracker entries Yes Yes Installed per Organization. Time entries live on the work item, so they follow the work item if it is moved between projects.

The core constraint to design around: An Area Path lives in exactly one Project. Anyone who needs to place a User Story into a P team’s Area Path must — at some point in the flow — be acting inside the P project, either directly (as a contributor) or indirectly (via an automation that runs on their behalf).

Recommended Solution

We recommend a two-layer intake model that combines a dedicated intake area inside P with a lightweight automation front door for teams that shouldn’t have direct P access. It works today, requires no custom code, and layers in optional enhancements as you grow.

🟦 Layer 1 — Intake Area in P

A dedicated P\_Intake Area Path with child nodes per receiving team. Every submitted story lands here first, regardless of source.

🟨 Layer 2 — Form + Flow Front Door

A Microsoft Form (or ADO work item template) feeding a Power Automate flow that creates the story in P with the right Area Path and a link back to the submitter.

🟩 Layer 3 — Triage & Allocation

Existing P triage meeting re-parents stories from _Intake to the owning team’s Area. Now your existing allocation model just works.

Why this works for : It respects the ADO Project boundary instead of fighting it. All P work — including externally-submitted work — ends up in the P project where your existing Area Path allocation model, 7Pace time tracking, Features, and Epics already operate. Submitters stay informed via the Related link back to their own project.

Pattern 1 — Dedicated Intake Area in P

The backbone of the solution. Works on day one with no automation. Other patterns build on top of this.

🟢 Intake Area Path Recommended Foundation

Create a first-class “front porch” inside the P project. Everyone who needs to send work to P ends up here, and P controls what happens next.

Area Path structure

P ├── TeamA # existing team Areas (unchanged) ├── TeamB ├── TeamC └── _Intake # NEW — note the leading underscore sorts it to the top ├── ForTeamA # optional: hint at target team ├── ForTeamB ├── ForTeamC └── Unassigned # default when submitter doesn't know who owns it

Ownership & process

  1. P owns the _Intake area. It is not a team backlog — it is a triage queue.
  2. Every story entering P from another project lands in _Intake with State = New.
  3. P runs a short standing triage (daily or 2×/week). On accepting a story, the triage owner:
    • Re-parents the story’s Area Path to the owning team, e.g. P\TeamA
    • Sets State = Active (or whatever your team uses)
    • Parents the story to the relevant Feature / Epic
    • Leaves any pre-existing “Related” link back to the submitter in place
  4. From that point, it is a normal P story. 7Pace, rollup, sprint planning — unchanged.

What the intake queue looks like

A saved shared query in the P project is the easiest triage view. Pin it to the P project dashboard.

-- Shared Query: "P Intake Queue" SELECT [Id], [Work Item Type], [Title], [State], [Created By], [Created Date], [Tags], [Area Path] FROM workitems WHERE [Team Project] = 'P' AND [Area Path] UNDER 'P\_Intake' AND [State] IN ('New', 'Proposed') ORDER BY [Created Date] ASC

Pattern 2 — Power Automate “Front Door”

This is how submitters in other Azure DevOps projects (or business teams with limited ADO access) get their story into P without needing to know your Area Path taxonomy. Zero code — a Power Automate flow does the crossing for them.

Option A — Trigger from ADO

Submitter creates a User Story in their own project with a specific tag (e.g. send-to-p) or custom field. A flow watches for new work items with that tag and creates a mirror in P.

  • Best when submitters already live in ADO
  • Keeps a copy in the submitter’s own project for their visibility
  • Creates a Related link automatically between the two IDs

Option B — Trigger from a Form

Submitter fills out a Microsoft Form (or a Power Apps canvas). Flow creates the User Story directly in P with the correct Area Path and notifies the submitter via email / Teams.

  • Best for business teams and ad-hoc requesters
  • No ADO license required for the submitter (Stakeholder license OK to view later)
  • Captures structured fields (priority, target team, justification)

Flow: “Create P Intake Story” (Option A example)

1

Trigger

When a work item is created — Organization = contoso, Project = Business-Ops, Work Item Type = User Story.

2

Condition

Only continue if Tags contains send-to-p. This keeps the flow opt-in and avoids runaway duplication.

3

Resolve target Area Path

Read a custom field (Custom.TargetPTeam) or parse tags like p-team:TeamA. Build the Area Path expression: P\_Intake\ForTeamA.

4

Create a work item in P

Action: Azure DevOps — Create a work item. Organization = contoso, Project = P, Type = User Story, Title = source title, Description = source description + “Submitted from [link to source WIT]”, Area Path = resolved value, Iteration Path = P, Tags = intake; from-business-ops.

5

Link the two work items

Action: Azure DevOps — Add a link. Link the source WIT to the new P WIT with relationship type System.LinkTypes.Related. The submitter keeps visibility in their project; the P WIT is the system of record for delivery.

6

Notify & close the loop

Post an adaptive card to a Teams channel, email the submitter with the P WIT ID, and (optionally) add a comment on the source WIT: “Intake created: P #12345.”

REST API payload (if you ever want to replace Power Automate with code)

# POST to create a User Story in P with the _Intake Area Path POST https://dev.azure.com/contoso/P/_apis/wit/workitems/$User%20Story?api-version=7.1 Content-Type: application/json-patch+json Authorization: Bearer <PAT or OAuth token> [ { "op": "add", "path": "/fields/System.Title", "value": "[Intake] Add SSO for claims portal" }, { "op": "add", "path": "/fields/System.AreaPath", "value": "P\\_Intake\\ForTeamA" }, { "op": "add", "path": "/fields/System.IterationPath", "value": "P" }, { "op": "add", "path": "/fields/System.Tags", "value": "intake; from-business-ops" }, { "op": "add", "path": "/fields/System.Description", "value": "Submitted from Business-Ops #4472 ..." }, { "op": "add", "path": "/relations/-", "value": { "rel": "System.LinkTypes.Related", "url": "https://dev.azure.com/contoso/_apis/wit/workItems/4472", "attributes": { "comment": "Origin intake request" } }} ]

No custom dev needed. The Power Automate connector covers all of this out of the box — create work item, update work item, add link, get work item details, get query results. Use the REST payload only if you want to embed intake into an internal portal or a bot.

Pattern 3 — Native “Move to P” (Lightweight Alternative)

If you don’t want Power Automate yet, Azure DevOps has a native feature that can carry you a long way: Move work item to another project. This is the simplest possible workflow and worth understanding as a fallback.

How it works

  1. Submitter creates a User Story in their own project like usual.
  2. They (or a P delegate with permissions on both sides) open the work item, click the … menu, and choose Move to team project.
  3. Select target project = P. Set Area Path = P\_Intake. Confirm.
  4. The work item is now a P work item, retains its comments, history, and attachments, and the old ID redirects to the new one.
  5. P triage proceeds as in Pattern 1.

When to choose this

  • You want zero automation and are willing to ask a P admin to click a button for each intake.
  • Volume is low (< 5 intakes per week).
  • Submitters understand they are handing off the story (they won’t see it in their own project’s backlog anymore unless you also create a Related link back).

Watch-outs (verified against Microsoft Learn)

  • Child items don’t move. Per Microsoft Learn: “Child work items don’t get moved and remain in the origin project, but the Parent-Child links remain in place.” Plan carefully when a story has subtasks.
  • Inherited process required. The Move work items out of this project permission requires the Inherited process model. On-premises XML processes don’t support this feature.
  • Compatible processes only. Custom fields that don’t exist in P’s process are stripped. State and Reason reset to the target type’s defaults; a comment is auto-added to Discussion and an entry to History.
  • Permissions needed: Move work items out of this project on the source and Edit work items in this node on P\_Intake. Stakeholder-license users cannot use this feature in private projects.
  • No backlog presence in source. The original ID URL redirects but the item no longer appears in the source team’s backlog. If submitters expect ongoing visibility, keep a tracking WIT in their project and add a Related link.
  • 7Pace entries travel with the work item. Historic time records remain attached after the move and roll up under P’s reports.

Implementation — Week-By-Week Plan

A pragmatic rollout that delivers value on day one and layers in automation and governance as the volume grows.

Week 1 — Foundation (Pattern 1)

🔧 Configuration

  • Create Area Path P\_Intake with child nodes per P team
  • Add a default child node P\_Intake\Unassigned
  • Configure the P Intake Triage team (or reuse P default team) with P\_Intake as its default Area Path
  • Create a shared query: P Intake Queue (see Pattern 1)
  • Pin the query to the P project dashboard
  • Add a work item template named “Intake — User Story” that pre-fills Area Path and a description template

👥 Access & Process

  • Add members of other IT team projects to a new Azure DevOps security group: P Intake Submitters
  • Grant that group Contribute on P\_Intake only — deny on other Area Paths
  • Deny Delete work items in this node to keep the intake record intact
  • Publish a one-page Wiki page: “How to submit work to P” with screenshots
  • Schedule a 15-minute daily P triage (or 2× per week to start)

Week 2–3 — Front Door (Pattern 2)

📝 Microsoft Form

  • Create form P Work Intake Request in Forms or SharePoint
  • Required fields: Title, Business justification, Target P team (dropdown), Requested by, Desired date, Priority
  • Share the form link in Teams / intranet

⚡ Power Automate Flow

  • Trigger: When a new response is submitted
  • Action: Azure DevOps — Create a work item in P, Type = User Story, Area Path mapped from form answer
  • Action: Add a comment with submitter email + form response link
  • Action: Send an email to submitter with the new P work item URL
  • Action (optional): Post adaptive card to P Triage Teams channel

Week 4+ — Governance & Insights

📊 Visibility

  • Build a Delivery Plan in ADO showing all P teams plus an Intake swim lane
  • Add a dashboard widget: Intake aging (stories > 7 days in _Intake)
  • Add chart: intakes per source project / per target team

🛡️ Guardrails

  • Add an ADO rule: when State moves out of New, require Area Path to no longer be under _Intake
  • Add a rule: intake stories require a “Business Value” or “Requested By” field
  • Service hook to alert if an intake is older than SLA (e.g. 3 days untriaged)

🧭 Process customization

  • If volume warrants it, add a custom WIT Intake Request to your inherited process
  • States: New → Triage → Accepted → Rejected
  • On Accepted, flow auto-spawns the real User Story and links it

Permissions Matrix

Getting the permissions right is what makes this feel seamless to submitters. Grant the minimum necessary and let the Area Path boundary do the enforcement.

License note: Users with a free Stakeholder license can view and comment on work items in a private project but cannot create or edit most fields. If business users will submit via ADO directly, they need at least a Basic license. The Microsoft Form + Power Automate path avoids this entirely because the service account writes on their behalf.

Enterprise Governance, Auditability & Compliance

For an enterprise deployment, intake workflow design is only half the story. The other half is proving to security, risk, and compliance stakeholders that every external request is captured, attributable, and retained for audit. The following controls are directly supported by Azure DevOps.

🛡️ Enable organization-level audit logging

Azure DevOps Auditing captures permission changes, resource deletions, branch policy changes, and log access — 90-day retention by default.

  • Turn on Log Audit Events in Organization Settings → Policies
  • Only Project Collection Administrators can access by default
  • Grant explicit View audit log permission to your compliance team
  • Available only for Microsoft Entra ID-backed organizations

🔄 Stream to SIEM for long-term retention

For retention beyond 90 days and enterprise incident response, Audit Streaming pushes events to Splunk, Microsoft Sentinel, or any SIEM.

  • Required for SOX / SOC 2 / ISO 27001 audit trails
  • Supports cloud and on-premises Splunk (allowlist IP ranges)
  • Every intake create / move / state change is captured
  • Correlate with Microsoft Entra sign-in logs for full attribution

7Pace Timetracker Considerations

the existing 7Pace investment is a key reason we recommend the work ultimately lives in the P project. Here’s how each pattern interacts with 7Pace.

  • Installation scope7Pace Timetracker is installed at the Organization level. Time entries are always attached to a work item, not to a project directly.
  • Pattern 1 (Intake Area)All time is logged against the P work item from the moment it exists. No change to reporting, timesheet, budgets, or approval workflows.
  • Pattern 2 (Form + Flow)Same as Pattern 1 — the flow creates the work item directly in P. 7Pace picks it up automatically.
  • Pattern 3 (Native move)Time entries travel with the work item when it is moved. Post-move, all further tracking continues normally. Historic entries stay attached and roll up under the new P project’s reports.
  • Anti-pattern to avoidDo not duplicate work items (one “delivery” story in P + one “tracking” story in the submitter project) for the same unit of work. Time will get logged against the wrong item and 7Pace reports will split. Use a single P work item with a Related link back to the requester for visibility.

FAQ & Gotchas

The questions that typically come up when rolling this out to other teams.

Can a work item have an Area Path from a different project?

No. Area Paths are scoped to their owning project. The story must live in the project that owns the Area Path. This is the constraint that shapes the entire solution.

Can submitters see the P story in their own project’s backlog?

Not directly — it’s a P work item. Use the Related link + a cross-project query saved in their project to give them a backlog view that includes their linked P stories.

What if two business teams want to submit the same story?

Merge them at triage. Keep one P story as the canonical item and add both source stories as Related links. This keeps 7Pace clean and avoids double-counting capacity.

Do we need to customize our process?

Not to start. The whole recommendation works on the out-of-the-box inherited process. Process customization (custom WIT for Intake Request, custom fields) becomes worthwhile once volume is high and you want structured metrics.

What about Epics and Features for intake work?

Intake stories enter without a parent. At triage, parent them to an existing Feature (if the work is part of an in-flight initiative) or to an “Unplanned / Intake” Feature under each team so it rolls up correctly.

How do we handle rejections?

Close the P intake story with State = Removed and a reason in the comment. The Related link back to the submitter’s work item makes the trail obvious. If using Pattern 2, the flow can optionally email the submitter with the rejection reason.

Can we automate the re-parent step?

Yes — an ADO rule or a secondary Power Automate flow can move a story from _Intake to a team Area based on a field (e.g. Assigned Team dropdown). Start manual to learn the patterns, then automate the steady-state rules.

Does this work with ADO Server / on-premises?

The Area Path / cross-project link / move features are identical. Power Automate cloud triggers won’t reach on-prem; use Service Hooks or the REST API with a self-hosted runner instead.
