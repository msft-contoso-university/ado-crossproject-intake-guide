Scene 1 of 7

One central project, many contributing teams

Inside the contoso organization, the P project is the single backlog of record. Other Azure DevOps projects — business units, IT teams, vendors — submit work into P rather than holding their own copies. Microsoft explicitly recommends using a single project to support your organization or enterprise.

  • Central P project owns the canonical User Story
  • Contributing projects use Remote Related links to keep their own context
  • Area Paths inside P route work to the right delivery team

Source: Microsoft Learn — Use a single project

Scene 2 of 7

Pick the intake pattern that fits

Three patterns are viable. The recommended one keeps the canonical work in P, because moving work items between projects loses associated history. Click a tab to see how each pattern shapes the topology and the trade-offs.

  • Canonical User Story is created directly in P
  • Contributing project links its own item via Remote Related
  • 7Pace, history, audit, and rollups all stay clean
  • Each project keeps a copy; Power Automate keeps fields in sync
  • Higher operational cost, duplicate state, drift risk
  • Use only when contributing project must own the work item
  • All projects share one Inherited process (P-Standard)
  • Consistent fields and rules everywhere; no field translation
  • Requires governance over who can change the process

Source: Move work items, loss of history · Link types incl. Remote Related

Scene 4 of 7

Identity & least-privilege access

Contributors authenticate through Microsoft Entra ID. Group membership flows from Entra into Azure DevOps; the right project, team, and Area-Path permissions follow automatically. JML (Joiner / Mover / Leaver) is centralized.

  • Entra group P-Intake-Contributors → ADO group Project Contributors
  • Stakeholder access for view-only / submit-only personas
  • Area-Path-scoped permissions for delivery teams

Scene 5 of 7

Roll up reporting in Power BI

Because the canonical work lives in P, the org-scoped Analytics OData endpoint pulls one consistent dataset. Power BI renders lead time, cycle time, time-in-state, and a CFD — with no manual reconciliation.

  • Org endpoint: https://analytics.dev.azure.com/contoso/_odata/v4.0-preview
  • Filter by Project/ProjectName eq 'P' and Area Path
  • One semantic model, refreshed nightly

Governance, audit, and guardrails

Every change is observable. Inherited process rules enforce data quality on the way in; service hooks drive notifications and SLA escalations; the audit log retains 90 days and can stream to a SIEM.

  • Required fields and transition rules at the process level
  • Service hook fires on WorkItemCreated → Teams + Power Automate
  • Audit events streamed to the enterprise SIEM

Source: Auditing in Azure DevOps · Service hooks

Scene 7 of 7

Phased rollout you can defend

A four-phase plan lands the program in roughly one quarter. Each phase has clear exit criteria so leadership can pause, expand, or roll back with confidence.

  • Foundation — process, Area Paths, Entra groups
  • Pilot — one contributing team, end-to-end flow
  • Scale — onboard remaining teams; bulk-import backlog
  • Optimize — KPIs, AI-assisted triage, dashboards
